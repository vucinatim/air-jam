## Purpose

This document defines the intended long-term documentation model for Air Jam.

It exists to keep documentation useful as the framework grows, while avoiding two bad outcomes:

1. docs becoming a second drifting implementation
2. “machine docs” turning into a parallel low-signal documentation product

The right model is simple:

1. code is the exact technical truth
2. human-written docs explain concepts, contracts, patterns, and product boundaries
3. machine-readable docs stay thin and exist only to improve navigation, retrieval, and future Studio integration

## Core Position

Air Jam should not build a giant generated-docs system.

It should also not rely on code alone to explain the framework.

The correct balance is:

1. **code-first for exactness**
2. **human docs for meaning**
3. **generated structure for accessibility**

This gives the best long-term result for:

1. framework users
2. maintainers
3. coding LLMs such as Codex, Claude, and Cursor
4. future Air Jam Studio workflows

## Documentation Layers

Air Jam should treat documentation as three distinct layers.

### 1. Code Truth

Code is the canonical source for:

1. exported APIs
2. types
3. runtime behavior
4. tests and edge cases
5. package boundaries
6. what actually ships

This includes:

1. source code
2. type definitions
3. tests
4. templates
5. examples

This layer answers:

1. what exists
2. what the signatures are
3. what the implementation really does

### 2. Human Docs

Human-written docs are the canonical source for:

1. concepts
2. system boundaries
3. ownership rules
4. canonical usage patterns
5. anti-patterns
6. migration guidance
7. release and product decisions

This layer answers:

1. why the system is shaped this way
2. which approach is canonical
3. what developers should do
4. what developers should avoid

Human docs must stay concise, opinionated, and explicit.

Air Jam should prefer fewer stronger docs over larger weaker docs.

### 3. Machine Layer

Machine-readable docs are still useful, but only as a **thin access layer**.

They should not try to replace either code or human prose.

Their job is:

1. help agents find the right docs quickly
2. expose stable metadata and routing
3. support search and retrieval
4. support future Studio ingestion and indexing

They should not become:

1. a second full docs system
2. generated explanatory prose
3. an attempt to encode the whole framework in one machine-only format

## What Should Be Written Manually

These should remain human-written:

1. conceptual guides
2. architectural explanations
3. contract explanations
4. usage recommendations
5. migration notes
6. release notes
7. agent guidance pages

These are valuable specifically because they express intent and judgment, not just inventory.

## What Can Be Generated

Generation is useful only where the source is already formal and reliable.

Good candidates:

1. page manifests
2. heading indexes
3. search indexes
4. page metadata bundles
5. API inventory tables when driven directly from exported source
6. version availability maps

Bad candidates:

1. autogenerated conceptual docs
2. autogenerated architecture prose
3. generated “best practices” text
4. generated migration guidance

The rule is:

1. generate structure
2. do not generate meaning unless the meaning is already explicit in formal source

## Public Docs Categories

Public Air Jam docs should stay organized into a small set of document types.

### 1. Concepts

Examples:

1. what Air Jam is
2. the three-lane model
3. host/controller mental model
4. standalone vs platform vs arcade

### 2. Guides

Examples:

1. quick start
2. add networked state
3. send haptics
4. self-host a game
5. publish a game
6. embed in Arcade

### 3. Reference

Examples:

1. hooks
2. signals
3. env vars
4. route expectations
5. package-level API surfaces

### 4. Contracts

Examples:

1. ownership rules
2. replay/reconnect rules
3. protocol invariants
4. bridge invariants
5. authoritative state boundaries

### 5. Migration / Release Notes

Examples:

1. what changed in a release
2. how to upgrade existing projects
3. whether an old pattern is still canonical

This category is especially important once the framework has external adoption.

## Internal Docs vs Public Docs

Air Jam should continue to separate:

1. **internal repo docs**
2. **public framework docs**

### Internal Repo Docs

These live in root `docs/`.

They should contain:

1. architecture direction
2. planning docs
3. release process
4. implementation trackers
5. maintainer-facing system decisions

### Public Framework Docs

These are the docs shown on the platform site.

They should contain:

1. user-facing framework concepts
2. guides
3. reference
4. contracts
5. migration and version guidance

The two sets should be linked, but not merged into one giant mixed audience.

## Target Structure

Air Jam should treat documentation structure as part of architecture, not just content placement.

The long-term goal is one clean content model with clear ownership.

### Preferred Long-Term Shape

The preferred future-proof structure is:

```text
docs/
  framework-paradigm.md
  documentation-architecture.md
  development-loop.md
  release-workflow.md
  implementation-plan.md
  plans/
  archive/
  internal/
    maintainers/
    platform/
    studio/

content/
  docs/
    concepts/
    guides/
    reference/
    contracts/
    migrations/
    for-agents/
  generated/
    docs-manifest.json
    docs-search-index.json
    docs-headings.json

apps/platform/
  src/app/docs/...
  src/app/llms.txt/route.ts
  src/app/docs-manifest/route.ts
  src/app/docs-search-index/route.ts
```

### Why This Shape

This structure creates one clean separation:

1. `docs/` is for repo and maintainer truth
2. `content/docs/` is for public framework docs content
3. `content/generated/` is for machine-readable generated artifacts
4. `apps/platform/` is the presentation layer that renders and serves those docs

That is the most extensible model because:

1. docs content is not trapped inside the web app folder
2. the platform stays a renderer and delivery layer
3. Studio can consume the same content and generated metadata later
4. search and agent endpoints can be generated from one canonical public docs source

## Acceptable Current Shape

The current implemented structure is acceptable and is now the canonical baseline:

1. root `docs/` for internal architecture and planning
2. `content/docs/` for canonical public docs content
3. `apps/platform/src/features/docs/` for the docs loader, registry, and machine-surface helpers
4. `apps/platform/src/app/docs/` as a rendering layer, not the source of truth
5. `llms.txt`, `/docs-manifest`, and `/docs-search-index` as a thin machine entry layer

This is the right direction.

The main remaining docs-system improvement is reducing manual source registration inside the extracted content adapter.

## Recommended Migration Path

Air Jam should not do a docs restructure for its own sake.

The clean path is incremental:

## Migration Goal

Air Jam should be willing to implement this documentation architecture in phases.

But the goal is **not** a permanent hybrid system.

The goal is:

1. phase the implementation carefully
2. complete the migration fully
3. remove superseded structure once the new system is proven
4. avoid keeping two documentation systems alive indefinitely

In other words:

1. phased rollout is good
2. permanent duplication is not

If Air Jam moves to the new docs system, it should eventually:

1. fully cut over to the canonical public docs source
2. fully cut over to canonical metadata-driven navigation and machine access
3. archive or delete the old paths that are no longer part of the intended system

This should be treated like any other architecture migration:

1. incremental while in progress
2. purged when complete

### Phase 1. Metadata And Ownership First

Do this before moving folders:

1. define page metadata for public docs
2. define page types: concept, guide, reference, contract, migration, agent
3. define ownership rules for internal vs public docs
4. generate a docs manifest and heading index from the existing public docs source

This gives the architecture without immediate churn.

This phase should end with explicit contracts, not just informal agreement.

### Phase 2. Public Docs Source Extraction

This is now implemented:

1. public docs source moved from `apps/platform/src/app/docs/` into `content/docs/`
2. the platform is now responsible for routing, rendering, search UI, and delivery
3. the old handwritten docs index was removed in favor of the docs feature module

This is the real structural cleanup step.

### Phase 3. Studio / Agent Reuse

Once content and metadata are canonical:

1. let Studio consume the same generated manifest and heading index
2. keep `llms.txt` and `for-agents` as lightweight human-curated entrypoints
3. avoid creating a separate Studio-only docs format

### Phase 4. Full Cutover And Purge

Once the new system is stable:

1. remove deprecated docs-source paths from the platform app
2. remove duplicated manual navigation structures that are no longer canonical
3. update docs tooling and contributor guidance to point only at the new system
4. archive any transitional notes that are no longer active

The migration is only complete when the old system is no longer the hidden fallback.

## What Should Be Defined Before Migration Starts

Yes, Air Jam is close to being ready to move to the new docs system.

But a few things should be defined first so implementation stays clean.

These should be decided before starting the real migration work:

### 1. Canonical Public Docs Source Location

Decide the final source location for public docs content.

Recommended:

1. `content/docs/`

Do not start migration work until this is explicit.

### 2. Page Metadata Schema

Define the exact metadata fields every public page should support.

Recommended minimum:

1. `title`
2. `description`
3. `docType`
4. `keywords`
5. `sinceVersion`
6. `lastVerifiedVersion`
7. `stability`
8. `audience`

### 3. Public Docs Taxonomy

Freeze the document-type taxonomy before moving content.

Recommended:

1. `concept`
2. `guide`
3. `reference`
4. `contract`
5. `migration`
6. `agent`

### 4. Machine Surface Scope

Define the minimal machine-readable surface that will exist after migration.

Recommended minimum:

1. `llms.txt`
2. `for-agents`
3. docs manifest JSON
4. heading/search index JSON

This avoids overbuilding.

### 5. Ownership And Contribution Rule

Define where contributors should add:

1. internal maintainer docs
2. public framework docs
3. migrations
4. agent-facing guidance

This should become an explicit contributor rule, not tribal knowledge.

### 6. Versioning Rule

Decide the docs versioning rule before implementation.

Recommended:

1. current stable docs live at `/docs`
2. page metadata carries version information
3. full versioned docs are added only for major compatibility divergence

### 7. Navigation Derivation Rule

Decide how much navigation stays manual versus metadata-derived.

Recommended:

1. page metadata is canonical
2. generated structures drive search/manifests
3. curated sidebar grouping remains allowed, but should derive from canonical page metadata where practical

## Readiness Assessment

Air Jam is ready to start this migration once the definitions above are written down explicitly.

The project does **not** need more paradigm work first.

What it needs is:

1. one small contract pass on metadata, taxonomy, and ownership
2. then disciplined implementation in phases

That is enough to start cleanly.

## Ownership Rules

To keep the structure clean, each docs class needs one owner.

### Root `docs/`

Owns:

1. internal architecture
2. implementation plans
3. release process
4. maintainer policy
5. product and system direction

Must not become:

1. the main public framework docs site source
2. a mixed bag of public tutorials and internal planning

### Public Docs Content

Owns:

1. framework concepts
2. user guides
3. reference
4. contracts
5. migrations
6. agent-facing public guidance

Must not become:

1. a dumping ground for maintainer planning
2. a mirror of every README in the repo

### Platform App

Owns:

1. rendering
2. layout
3. navigation UI
4. search UI
5. machine-readable route delivery

Must not own:

1. the canonical public docs content model long-term
2. manually duplicated docs structure that can drift from content

## Metadata Model

Every public docs page should eventually expose metadata like:

1. `title`
2. `description`
3. `docType`
4. `keywords`
5. `sinceVersion`
6. `lastVerifiedVersion`
7. `stability`
8. `audience`

This metadata should drive:

1. navigation
2. search
3. manifests
4. future Studio retrieval

The page file should remain the source of meaning.
The metadata should make that meaning navigable.

## Machine-Readable Surface

Air Jam should keep a very small machine-readable documentation surface.

The intended shape is:

### 1. `llms.txt`

Purpose:

1. simple public entrypoint
2. top-level orientation
3. links to the best docs

This should stay short and stable.

### 2. `for-agents`

Purpose:

1. curated entry page for coding agents
2. links to core concepts, contracts, and guides
3. highlights canonical rules and common mistakes

This should be written by humans.

### 3. Docs Manifest

A small generated manifest should eventually exist.

It should contain metadata like:

1. title
2. url
3. section
4. description
5. keywords
6. headings
7. doc type
8. version metadata
9. last updated date

This is useful for:

1. docs search
2. external tools
3. Studio indexing
4. lightweight agent retrieval

### 4. Search / Heading Index

A generated heading-level index is worth having because it helps:

1. humans jump to the right section
2. agents retrieve section-level answers instead of whole-page guesses

This is structural generation, not prose generation.

## Why Machine Docs Still Matter

Machine docs are not needed because LLMs “cannot read code”.

They matter because:

1. agents often need fast orientation before deep code reading
2. external projects may only have installed packages, not the whole Air Jam repo
3. concept pages and contracts are often clearer than implementation files for framework intent
4. future Studio systems benefit from stable retrieval surfaces

So the right conclusion is:

1. **code remains primary**
2. **machine-readable docs remain useful**
3. **the machine layer should stay thin**

## Versioning Strategy

Versioning should be built into docs metadata early, but full multi-version docs should only be introduced when truly needed.

### Default Rule

1. `/docs` points to the latest stable documentation
2. each page should eventually carry version metadata
3. migrations and release notes cover most normal evolution

Recommended metadata:

1. `sinceVersion`
2. `lastVerifiedVersion`
3. `stability`: `stable | evolving | experimental`

### Full Versioned Docs

Full docs snapshots such as `/docs/v1/...` should be added only when:

1. a major version introduces meaningful compatibility divergence
2. current docs can no longer safely serve older projects with migration notes alone

This avoids premature version sprawl.

## Search And Navigation

Docs search should eventually be driven by canonical page metadata rather than only by manually maintained page lists.

The search system should support:

1. page-level discovery
2. heading-level discovery
3. keyword search
4. version-aware filtering later

The docs index should remain curated, but the searchable structure should be derived from the docs source where practical.

## Future Studio Compatibility

This docs architecture should support Air Jam Studio later without a rewrite.

That means:

1. Studio should be able to consume a small manifest/index layer
2. Studio should still rely on human docs and source code for real answers
3. docs should not be rewritten into a Studio-only format

The Studio model should be:

1. use machine-readable metadata for retrieval
2. use human docs for conceptual grounding
3. use source code and tests for exact behavior

## Non-Goals

Air Jam documentation should explicitly avoid:

1. a giant auto-generated prose docs system
2. duplicate explanations spread across many pages
3. a machine-docs layer that tries to restate the whole framework
4. premature full version snapshots for every release
5. mixing maintainer planning docs and public user docs into one indistinct set

## Practical Direction

The practical future-proof documentation model for Air Jam is:

1. keep human docs curated and opinionated
2. treat source code and tests as exact truth
3. generate only structure and metadata around the docs
4. keep a small agent-facing access layer
5. add lightweight version metadata now
6. delay full multi-version docs until the framework actually needs them

## Closeout Rule

When making documentation decisions, choose the option that:

1. keeps conceptual truth in human docs
2. keeps exact truth in code
3. avoids duplicate maintenance burden
4. makes future agent and Studio retrieval easier without inventing a second docs product
