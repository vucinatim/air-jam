# Air Jam Discoverability Vision

Last updated: 2026-05-07  
Status: launch-direction vision

## Purpose

This document defines the long-term discoverability vision for Air Jam.

It exists to keep one thing explicit:

Air Jam should become easy to find, easy to classify, easy to trust, and easy to retrieve without turning into a noisy SEO-shaped product.

Discoverability is not separate from product quality.
It is part of product clarity.

## Vision

Air Jam should become a real, recognizable thing on the public web.

That means:

1. humans should be able to understand what it is quickly
2. Google should be able to classify it with minimal ambiguity
3. npm and GitHub should reinforce the same product story
4. LLM agents should be able to find the correct docs, packages, and machine-readable surfaces without guessing
5. every public surface should point toward the same canonical understanding of the product

The goal is not to get attention through content volume.

The goal is to make Air Jam structurally obvious.

## Core Principle

Air Jam should improve discoverability through truth density, not content volume.

That means:

1. fewer pages with stronger jobs
2. one canonical explanation per concept
3. real examples instead of filler
4. machine-readable endpoints only when they reflect real maintained contracts
5. no thin landing pages, duplicate docs, or fake AI-targeted copy

If a page, endpoint, or metadata surface would not be worth keeping without search engines, it should probably not exist.

## Public Product Shape

Air Jam should present through a small set of clear public nouns:

1. `Air Jam` as the umbrella product
2. `@air-jam/sdk` as the game authoring framework
3. `@air-jam/server` as the realtime runtime/server lane
4. `@air-jam/mcp-server` as the agent and devtools integration lane
5. `create-airjam` as the project creation CLI

These should be the main public entrypoints across:

1. the website
2. the docs
3. package metadata and READMEs
4. GitHub repository metadata and releases
5. examples and launch material
6. machine-readable endpoints

Air Jam should not present as a vague platform blob.
It should present as one coherent system with a small number of obvious public surfaces.

## What Discoverability Means For Air Jam

For Air Jam, discoverability is not just about rankings.

It means:

1. the product can be found
2. the product can be understood
3. the product can be trusted
4. the product can be installed
5. the product can be integrated
6. the product can be retrieved by agents cleanly

This matters because the long-term Air Jam vision depends on both humans and agents operating through explicit contracts.

If the public surface is fragmented, vague, or contradictory, that weakens the product even if the internals are good.

## Canonical Surface Philosophy

Each important public concept should have one obvious canonical home.

Examples:

1. one homepage that says what Air Jam is
2. one docs path that says how to use it
3. one primary page per public package
4. one architecture path that explains the system
5. one agent-facing/devtools path that explains machine-usable surfaces
6. one release/update lane that records meaningful changes

The point is not to minimize information.

The point is to prevent drift, duplication, and ambiguity.

## Search And Agent Retrieval

Search discovery and agent retrieval are closely related.

Both improve when Air Jam has:

1. stable canonical URLs
2. explicit package roles
3. clear metadata
4. crawlable server-rendered docs
5. accurate structured data
6. trustworthy examples
7. machine-readable indexes that point to real maintained surfaces

`llms.txt` and similar endpoints are useful only if they reflect a truthful public information architecture.

They should never become a second shadow docs system.

## Anti-Bloat Rule

Air Jam should reject discoverability tactics that make the product feel cheap.

That includes:

1. thin SEO pages
2. duplicate docs written for alternate wording only
3. content generated just to target keywords
4. fake “AI-optimized” copy
5. dead machine-readable endpoints
6. package or docs surfaces that overpromise unstable internals

The right discoverability strategy is not more noise.

It is better structure, better naming, better metadata, better examples, and better canonical ownership.

## Long-Term Outcome

The ideal end state is:

1. Air Jam is the clearest answer to its product category
2. the public web reflects the real architecture of the system
3. users know where to start without confusion
4. agents can retrieve the correct surfaces with minimal ambiguity
5. discoverability grows as a side effect of product clarity, not content spam

## Design Rule

When choosing how to make Air Jam more visible, prefer the option that:

1. makes the product more canonical
2. makes the public surface more explicit
3. reduces duplication
4. keeps maintenance ownership clear
5. helps both humans and agents understand the same underlying system

If a discoverability tactic increases noise, splits the story into too many surfaces, or creates a public picture that does not match the real product, it is the wrong direction.
